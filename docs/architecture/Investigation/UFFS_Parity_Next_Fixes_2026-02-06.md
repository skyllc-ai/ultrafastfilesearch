## 2) Offline scan – the remaining 20 mismatches have a single signature

Here are some of the offline mismatches (from your report), showing just the deltas:

| Path (directory) | C++ Size | Rust Size | Delta |
|---|---:|---:|---:|
| `...\wow64_microsoft-windows-s..r\n\` | 48 | 56 | +8 |
| `...\assembly\GAC_MSIL\...` | 53738 | 53794 | +56 |
| `...\netfx-aspnetmmcext_...` | 254380 | 254444 | +64 |
| `...\coreuicomponents_...` | 65712 | 65776 | +64 |
| `...\WindowsApps\...\MyASUS\` | 10522745 | 10522757 | +12 |

These deltas are **small** and look like “we are counting a small resident blob that C++ excludes”.

### Strong hypothesis: Rust is still counting `$I30:$BITMAP` bytes in the directory “size”

On NTFS directories, the “directory index” consists of:

- `$INDEX_ROOT` (resident)
- `$INDEX_ALLOCATION` (non-resident)
- `$BITMAP` (often resident for modest indexes)

Your current Rust parsing path treats these as a single logical “directory index stream” (good), but it also **adds `$BITMAP` resident `value_length` into the same stream size**.

C++’s reference behavior (based on the deltas) appears to be:

- include `$INDEX_ROOT` size
- include `$INDEX_ALLOCATION` **allocated** size in “Size”
- **exclude `$BITMAP`** from “Size”

That exactly explains deltas like +8, +12, +56, +64.

This is also consistent with the original “Fix 2” intent (“exclude Bitmap”) — the current code path does not actually exclude it.

## 3) Fix #7 (new): Truly exclude `$I30:$BITMAP` from size accounting

### Where to fix
`crates/uffs-mft/src/cpp_types.rs` in `parse_stream()` / `update_stream_sizes()`.

Your current logic:

- detects `$I30` via `(type ∈ {IndexRoot, IndexAllocation, Bitmap}) && name == "$I30"`
- merges all those attributes into the same directory stream
- **always** adds resident `value_length` into `record.first_stream.size.length`

That is why `$BITMAP` inflates directory size.

### Minimal patch (safe, surgical)

**Change 1:** add an early skip in `parse_stream()` when the attribute is `$BITMAP` and it’s part of `$I30`.

```diff
diff --git a/crates/uffs-mft/src/cpp_types.rs b/crates/uffs-mft/src/cpp_types.rs
index <old>..<new> 100644
--- a/crates/uffs-mft/src/cpp_types.rs
+++ b/crates/uffs-mft/src/cpp_types.rs
@@ fn parse_stream(&mut self, record_idx: usize, attr_data: &[u8], attr_total_size: usize) {
         let is_dir_index = matches!(
             type_code,
             x if x == AttributeType::IndexRoot.to_u32()
                 || x == AttributeType::IndexAllocation.to_u32()
                 || x == AttributeType::Bitmap.to_u32()
         ) && stream_name == b"$I30";

+        // Parity: C++ does NOT count $I30:$BITMAP toward directory "Size"/tree metrics.
+        // Bitmap is metadata (used/free slots in the directory index), not part of directory byte size.
+        if is_dir_index && type_code == AttributeType::Bitmap.to_u32() {
+            return;
+        }
```

That alone should remove the +8/+12/+56/+64 deltas.

## 4) Fix #2 (confirm / harden): `$I30:$INDEX_ALLOCATION` “Size” must use allocated size

Even if your current datasets happen to have `data_size == allocated_size` in many cases, you should implement this explicitly, because **this is the exact semantic difference** that originally caused huge mismatches.

### Where to fix
`update_stream_sizes()` in `cpp_types.rs`.

### Patch (make length use allocated size for `$I30` index allocation)

This requires passing `type_code` + `is_dir_index` into `update_stream_sizes()`.

```diff
diff --git a/crates/uffs-mft/src/cpp_types.rs b/crates/uffs-mft/src/cpp_types.rs
index <old>..<new> 100644
--- a/crates/uffs-mft/src/cpp_types.rs
+++ b/crates/uffs-mft/src/cpp_types.rs
@@
-    fn update_stream_sizes(
-        index: &mut CppMftIndexPipeline,
-        attr_data: &[u8],
-        frs_base: u32,
-        is_non_resident: bool,
-    ) {
+    fn update_stream_sizes(
+        index: &mut CppMftIndexPipeline,
+        attr_data: &[u8],
+        frs_base: u32,
+        is_non_resident: bool,
+        is_dir_index: bool,
+        type_code: u32,
+    ) {
         let record = &mut index.records_data[frs_base as usize];

         if is_non_resident {
             let non_res = unsafe {
                 &*(attr_data[size_of::<AttributeRecordHeader>()..].as_ptr()
                     as *const NonResidentAttributeData)
             };

-            let allocated = i64_to_u64_filetime(non_res.allocated_size);
-            let data_size = i64_to_u64_filetime(non_res.data_size);
+            let allocated = i64_to_u64_filetime(non_res.allocated_size);
+            let mut data_size = i64_to_u64_filetime(non_res.data_size);
+
+            // Parity: for $I30:$INDEX_ALLOCATION, C++ uses allocated_size as the "Size".
+            if is_dir_index && type_code == AttributeType::IndexAllocation.to_u32() {
+                data_size = allocated;
+            }

             let bulkiness = allocated;

             record.first_stream.size.allocated =
                 record.first_stream.size.allocated.saturating_add(FileSizeType::new(allocated));
             record.first_stream.size.bulkiness =
                 record.first_stream.size.bulkiness.saturating_add(FileSizeType::new(bulkiness));
             record.first_stream.size.length =
                 record.first_stream.size.length.saturating_add(FileSizeType::new(data_size));
         } else {
             let resident = unsafe {
                 &*(attr_data[size_of::<AttributeRecordHeader>()..].as_ptr()
                     as *const ResidentAttributeData)
             };

             let data_size = u64::from(resident.value_length);

             record.first_stream.size.length =
                 record.first_stream.size.length.saturating_add(FileSizeType::new(data_size));
         }
     }
```

Now update the two call-sites inside `parse_stream()`:

```diff
@@
-            Self::update_stream_sizes(index, attr_data, frs_base, is_non_resident);
+            Self::update_stream_sizes(index, attr_data, frs_base, is_non_resident, is_dir_index, type_code);
             return;
@@
-        Self::update_stream_sizes(index, attr_data, frs_base, is_non_resident);
+        Self::update_stream_sizes(index, attr_data, frs_base, is_non_resident, is_dir_index, type_code);
```

Combined with the `$BITMAP` skip, this should eliminate the remaining offline mismatches.

## 6) Live scan “regression”: treat it as a separate issue (tooling / run integrity)

A live scan that only produces **871k** rows when C++ and offline both produce **2.221M** is **not** a “small parity mismatch”. It’s almost certainly one of:

- The live scan output file is **partial/truncated** (scan terminated early, or file overwritten mid-run, or analysis read it early).
- The live scan output was produced by a **different binary** than the one you think (old `uffs.exe` path).
- The analyzer is **parsing the live CSV incorrectly** (schema/header mismatch, early parse failure, etc.).
- The script is scanning a different target (e.g., wrong root, different drive letter).

### Concrete “no excuses” integrity checks to add

#### A) Write CSV outputs atomically
Change the CLI so `--csv-output <path>` writes to `<path>.tmp` and renames to `<path>` only on success.

This eliminates partial files being treated as “valid runs”.

#### B) Add a `--build-id` / `--tripwire` command
Since strict CSV can’t contain a tripwire comment anymore, make the binary itself print a stable marker:

- `uffs.exe --build-id` prints `TRIPWIRE=<string>`
- or include `tripwire=<string>` in `uffs.exe --version`

Then the trial script can log it without needing external `strings.exe`.

#### C) Trial script: always log line counts
After each scan:

- count lines (`(Get-Content file | Measure-Object -Line).Lines`) or faster: `Get-Item file | Select Length` + `Select-String -List` sanity
- record expected range (millions for F:)

If Rust live lines are 800k, the script should fail fast and not proceed to parity compare.

- Rust size larger by small “blob” sizes (8/12/56/64)

This is the cleanest “single remaining lever” consistent with the data.
